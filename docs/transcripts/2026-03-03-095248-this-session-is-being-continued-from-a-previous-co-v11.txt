‚óŹ Five things were blocking it, fixed one after another:

  1. Syntax error from our own patch ‚ÄĒ an earlier script to fix a dtype issue
  created malformed Python. Fixed the patch script to preserve proper
  indentation.
  2. Auto-quantization kicked in ‚ÄĒ ACE-Step looked at the 11 GB GPU and said
  "great, I'll use INT8 quantization!" That feature requires a newer GPU library
   (torchao) that doesn't support your Pascal card. Added a startup check: if
  Pascal GPU detected ‚Üí skip quantization entirely.
  3. Missing Python packages ‚ÄĒ the training UI imported matplotlib and diskcache
   which weren't installed. Added them.
  4. Two processes competing for VRAM ‚ÄĒ the startup script was launching both
  the Gradio UI and a REST API server, each loading the 4.5 GB AI model onto the
   GPU simultaneously. OOM crash. Switched to one process only, and used the
  smaller 0.6B language model (1.3 GB) instead of the 1.7B (3.5 GB).
  5. BF16 vs FP16 ‚ÄĒ ACE-Step assumed all CUDA GPUs support BF16 (brain float).
  Pascal doesn't have native BF16 hardware. Patched it to detect Pascal and use
  FP16 instead.

  The thing that made it all possible: even though PyTorch's prebuilt wheels
  don't explicitly compile for your GPU's exact variant (sm_61), CUDA has a
  backward compatibility rule where sm_60 code runs fine on sm_61 hardware. So
  we didn't need to compile PyTorch from source ‚ÄĒ just needed to get past all
  the software-level blockers above.
